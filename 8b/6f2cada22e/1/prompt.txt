## Blocking Issues

### B1. `fetchAllMessages` infinite loop / stale closure

**File:** `/Users/melagiri/Workspace/codeInsights/code-insights/dashboard/src/components/sessions/SessionDetailPanel.tsx`, lines 81-90

```typescript
const fetchAllMessages = useCallback(async () => {
    if (loadingAllMessages || !messagesQuery.hasNextPage) return;
    setLoadingAllMessages(true);
    let query = messagesQuery;
    while (query.hasNextPage) {
      await query.fetchNextPage();
      query = messagesQuery; // <-- BUG: reassigns to same stale closure reference
    }
    setLoadingAllMessages(false);
  }, [messagesQuery, loadingAllMessages]);
```

The `while` loop reassigns `query = messagesQuery` on each iteration, but `messagesQuery` is captured by closure and never changes within this invocation. After `fetchNextPage()` resolves, `messagesQuery.hasNextPage` still reflects the value at the time the callback was created. This will either:

1. Loop infinitely (if `hasNextPage` remains truthy in the closed-over reference), or
2. Exit after one iteration (if React Query updates the ref between awaits, which it does not -- `messagesQuery` is a new object reference on re-render, not a mutated object).

The most likely behavior is that `await query.fetchNextPage()` resolves, then `query = messagesQuery` still points to the same old object, so `query.hasNextPage` is still `true`, causing an infinite fetch loop until React Query eventually has no more pages to report.

**Recommendation:** Use a loop that checks via `fetchNextPage` return value, or use a ref to track the query object, or simply call `fetchNextPage` once per invocation and let the `useCallback` dependency on `messagesQuery` naturally re-create the callback. Example fix:

```typescript
const fetchAllMessages = useCallback(async () => {
    if (loadingAllMessages) return;
    setLoadingAllMessages(true);
    try {
      let result = await messagesQuery.fetchNextPage();
      while (result.hasNextPage) {
        result = await result.fetchNextPage();
      }
    } finally {
      setLoadingAllMessages(false);
    }
  }, [messagesQuery, loadingAllMessages]);
```

Or use `fetchNextPage`'s returned result directly, which is the updated query state.

### B2. `useEffect` with `matches.join(',')` as dependency -- React hooks rule violation

**File:** `/Users/melagiri/Workspace/codeInsights/code-insights/dashboard/src/components/chat/conversation/ConversationSearch.tsx`, lines 39-41

```typescript
useEffect(() => {
    onHighlightMessage(matches[matchIndex] ?? null);
  }, [matches.join(','), matchIndex]);
```

Two issues:

1. **Missing dependency: `onHighlightMessage`**. The eslint `react-hooks/exhaustive-deps` rule would flag this. If `onHighlightMessage` changes identity across renders (e.g., if the parent does not memoize it), the effect will be stale.

2. **`matches.join(',')` is an expression in the deps array.** While this is a known technique for array-value dependencies, it suppresses the exhaustive-deps lint rule and masks the fact that `matches` itself is also missing from the dependency array. This works but is fragile -- if two different match sets happen to produce the same joined string (unlikely but possible with IDs containing commas), the effect would not fire.

**Recommendation:** Use `JSON.stringify(matches)` or extract `matches` to a `useMemo` and reference it + `onHighlightMessage` explicitly:

```typescript
const matchIds = useMemo(() => 
  debouncedQuery
    ? messages.filter(m => m.content.toLowerCase().includes(debouncedQuery.toLowerCase())).map(m => m.id)
    : [],
  [messages, debouncedQuery]
);

useEffect(() => {
  onHighlightMessage(matchIds[matchIndex] ?? null);
}, [matchIds, matchIndex, onHighlightMessage]);
```

### B3. Union-find `find()` function can stack overflow on large linked chains

**File:** `/Users/melagiri/Workspace/codeInsights/code-insights/dashboard/src/components/insights/RecurringPatternsSection.tsx`, lines 44-48

**Also duplicated in:** `/Users/melagiri/Workspace/codeInsights/code-insights/dashboard/src/pages/InsightsPage.tsx`, lines 58-62

```typescript
function find(id: string): string {
      if (!parent.has(id)) parent.set(id, id);
      if (parent.get(id) !== id) parent.set(id, find(parent.get(id)!));
      return parent.get(id)!;
    }
```

This is a recursive union-find with path compression. For deeply linked insight chains (e.g., 1000+ sequential links), this will cause a stack overflow. While unlikely with current data sizes, if this is used on the Insights page with all insights loaded, the chain depth is bounded only by the data.

**Recommendation:** Convert to iterative path compression:

```typescript
function find(id: string): string {
  if (!parent.has(id)) parent.set(id, id);
  let root = id;
  while (parent.get(root) !== root) root = parent.get(root)!;
  // path compression
  let curr = id;
  while (curr !== root) {
    const next = parent.get(curr)!;
    parent.set(curr, root);
    curr = next;
  }
  return root;
}
```
