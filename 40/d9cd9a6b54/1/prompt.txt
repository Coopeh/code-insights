`★ Insight ─────────────────────────────────────`
**`INSERT OR REPLACE` vs `INSERT ... ON CONFLICT DO UPDATE`:**
- `INSERT OR REPLACE` = DELETE old row + INSERT new → **all** unspecified columns reset to defaults
- `INSERT ... ON CONFLICT DO UPDATE SET col=val` = true upsert → only updates specified columns, **preserves** unspecified ones
- SQLite's `ON CONFLICT` (also called "UPSERT") was added in SQLite 3.24 (2018), so `better-sqlite3` supports it

### Issues Found

**🟡 SUGGESTION — SQL query change: `au.analysis_type IS NULL` to `au.session_id IS NULL`**
(`cli/src/commands/insights.ts`, diff line in the SELECT query)

The old query used `AND au.analysis_type IS NULL` to detect unjoined rows. The PR changes this to `AND au.session_id IS NULL`. Both are semantically correct for a LEFT JOIN (any non-nullable column from the right table works), but `session_id IS NULL` is slightly more idiomatic for "no matching row found." **🟡 SUGGESTION — `_runner` as a public interface field is a mild code smell**
(`cli/src/commands/insights.ts:100`)

If the interface grows or becomes part of a public API contract, consider a separate internal overload or options type. Low priority — acceptable for now.

**🟡 SUGGESTION — Auto-analyze (1-2 sessions) silently swallows all errors**
(`cli/src/commands/insights.ts`, the `count <= 2` block)

The empty `catch {}` means if `validate()` succeeds but every analysis fails (e.g., claude CLI hangs or crashes), the user gets zero feedback. For 1-2 sessions this is likely fine (the intent is "opportunistic"), but consider at minimum a `--verbose` path in the future. Not blocking.

`--analyze` processes sessions one at a time in a `for` loop. This is the right call for `claude -p` which spawns a subprocess — parallelizing would risk resource exhaustion and interleaved output. No change needed.
