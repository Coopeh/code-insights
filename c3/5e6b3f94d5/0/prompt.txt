- No LLM code touched

### Prompt Quality Assessment

**FRICTION_CLASSIFICATION_GUIDANCE** (the core change)

The old guidance was structured as negative anchoring ("Do NOT use X for Y") which only steered the LLM away from one category (`tooling-limitation`) but gave no positive guidance for the other 14. The new guidance uses **contrastive definitions** — each category gets a one-line definition with a distinguishing signal word in caps (STRATEGY, KNOWLEDGE, STATE, FORGOT, etc.). This is a textbook prompt engineering improvement: contrastive pairs with semantic anchors dramatically reduce classification ambiguity.

### Token Efficiency

**Rating: 4/5** — Good, with one minor note.

The new guidance is slightly longer in raw characters but delivers far more information per token. The old version spent tokens on 4 negative examples for a single category. The new version covers all 9 categories with contrastive definitions in roughly the same token budget. Net: better information density.

One minor inefficiency: the `tooling-limitation` negative examples at the bottom (`Do NOT use "tooling-limitation" for: agent spawning failures...`) partially overlap with the contrastive definition above it. This is ~20 tokens of redundancy but is justified as belt-and-suspenders for the known catch-all problem. Not worth removing.

### Contrastive Guidance Quality

**Rating: 5/5** — Excellent.

The contrastive pairs are well-designed for LLM classification:

1. **wrong-approach vs knowledge-gap**: Strategy vs Knowledge — clean semantic boundary. "Tried rewriting instead of patching" (strategy) vs "used deprecated API" (knowledge). LLMs will distinguish these correctly.

2. **knowledge-gap vs stale-assumptions**: Knowledge vs State — subtler but the examples nail it. "Wrong function signature" (knowledge) vs "assumed package installed" (state). The key insight is that stale-assumptions is about the *environment state* being different from what was assumed, while knowledge-gap is about *technical facts* being wrong.

3. **context-loss vs repeated-mistakes**: These could be confused (both involve "doing something wrong again") but the definitions disambiguate well — context-loss is about *forgetting within a session*, repeated-mistakes is about *failing to learn from corrections*.

4. **scope-creep vs incomplete-requirements**: Both involve scope mismatch but from opposite directions — AI did too much vs user specified too little. Clear distinction.

5. **documentation-gap vs tooling-limitation**: External resource vs tool capability — straightforward.

**Potential confusion point** (minor): `stale-assumptions` clusters 6 legacy categories (missing-dependency, config-drift, stale-cache, version-mismatch, permission-issue, environment-mismatch). Some of these — particularly `permission-issue` — could arguably be `knowledge-gap` (the AI didn't know it needed sudo) rather than `stale-assumptions` (the AI assumed it had permissions). However, the current mapping is defensible: the AI *assumed* a state (has permissions) that was incorrect, which fits the "incorrect STATE assumption" definition. This is a judgment call and the current choice is reasonable.
