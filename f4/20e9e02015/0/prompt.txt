This works but is unusual. A separate function or early return pattern would be clearer:
```typescript
function getCollapsedPreview() { ... }
const collapsedPreview = !expanded ? getCollapsedPreview() : null;
```

**5. `iconColorClass` extraction is brittle**

In `InsightListItem.tsx`:
```typescript
const iconColorClass = colorClass.split(' ').find(c => c.startsWith('text-')) || 'text-muted-foreground';
```

This parses CSS class strings to extract color info, which is fragile â€” it depends on how `INSIGHT_TYPE_COLORS` formats its strings. If the constant changes format (e.g., uses Tailwind v4 syntax), this breaks silently. Consider adding a dedicated `INSIGHT_TYPE_TEXT_COLORS` constant or restructuring `INSIGHT_TYPE_COLORS` as an object with separate fields.

**6. Missing accessibility on InsightListItem expand/collapse**

The `<button>` in `InsightListItem.tsx` has `aria-expanded={expanded}` (good), but no `aria-controls` linking it to the expanded panel. For full ARIA disclosure pattern compliance, the expanded content `<div>` should have an `id` and the button should have `aria-controls` pointing to it.

**7. Inconsistent key usage in list rendering**

In `InsightCard.tsx` `SummaryContent` and `InsightListItem.tsx` `SummaryExpandedContent`, bullets use array index as key:
```typescript
<li key={i} ...>
```

Since these lists are static (not reordered or filtered), index keys are acceptable here, but if bullets could change between renders (e.g., live analysis updates), this could cause React reconciliation issues. Not critical but worth noting.

**8. `InsightMetadata` type uses camelCase for some fields**

The `InsightMetadata` interface in `types.ts` mixes naming conventions:
- snake_case: `root_cause`, `trade_offs`, `revisit_when`, `applies_when`, `rejected_because`, `linked_insight_ids`
- camelCase: `efficiencyScore`, `wastedTurns`, `antiPatterns`, `sessionTraits`, `potentialMessageReduction`, `messageIndex`, `whatWentWrong`, `suggestedRewrite`, `turnsWasted`

The codebase convention (documented in CLAUDE.md) is snake_case for dashboard types matching SQLite/API responses. The camelCase fields in the prompt_quality section may reflect the actual LLM output format, but the inconsistency is worth flagging. If the LLM generates camelCase, this is fine as-is â€” just document why the convention differs.

**9. `SessionRow` component is defined in the same file as `SessionsPage`**

`SessionRow` and `InsightCountBadge` are defined at the bottom of `SessionsPage.tsx`. These are substantial components (50+ lines each). For a page that's already 300+ lines, extracting them to `components/sessions/SessionRow.tsx` would improve maintainability. Not blocking, but worth considering.

**10. Cost color thresholds are hardcoded**

`getCostColor()` in `SessionsPage.tsx` uses hardcoded thresholds ($0.10, $0.50). These may not suit all users â€” a Claude Opus session could easily exceed $0.50. Consider making these configurable or at minimum documenting why these thresholds were chosen.
